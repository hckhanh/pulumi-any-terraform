# Contributing Guide

This document defines the coding standards, style conventions, and contribution practices for the `pulumi-any-terraform` project. All contributions should follow these rules unless otherwise noted.

## 1. General Code Style

- Favor clarity over brevity.
- Keep functions and methods small and focused.
- Avoid repeating logic; prefer shared helpers/utilities (see `tools/utils/`).
- Remove unused variables, imports, code paths, and files.
- ~95% of TypeScript in `packages/` is **auto-generated** by `pulumi-language-nodejs`. Do not edit these files by hand unless you are certain of what you are doing. Hand-written code lives in `tools/`, `.github/scripts/`, and `packages/*/scripts/`.

## 2. Project Structure

```
.
├── packages/         # Auto-generated Pulumi provider packages
│   └── <provider>/
│       ├── *.ts              # Generated TypeScript source
│       ├── scripts/
│       │   └── postinstall.js  # Build script (runs tsc)
│       └── package.json
├── tools/            # Hand-written Nx plugins (TypeScript)
│   ├── biome.ts
│   ├── build.ts
│   ├── linter.ts
│   ├── prettier.ts
│   └── utils/plugin.ts
├── docs/             # Fumadocs documentation site
└── .github/
    ├── scripts/      # CI automation scripts (JavaScript)
    └── workflows/    # GitHub Actions workflows
```

## 3. Naming Conventions

| Item         | Convention                                          | Example                                               |
| ------------ | --------------------------------------------------- | ----------------------------------------------------- |
| Variables    | `camelCase`                                         | `packageJson`, `latestVersion`                        |
| Functions    | `camelCase`                                         | `determineBumpType()`, `getEnv()`                     |
| Classes      | `PascalCase`                                        | `Plugin`, `BiomePlugin`, `Provider`                   |
| Constants    | `camelCase` or `UPPER_SNAKE_CASE`                   | `_packageRef`, `MAX_RETRIES`                          |
| Type imports | `import type`                                       | `import type { CreateNodesResult } from '@nx/devkit'` |
| File names   | `camelCase.ts` (generated), `kebab-case` (packages) | `postinstall.js`, `check-updates.js`                  |

## 4. Formatting Rules

Formatting is enforced automatically by **Biome** (JS/TS/JSON/CSS) and **Prettier** (YAML, Markdown, HTML).

| Rule            | Value                              |
| --------------- | ---------------------------------- |
| Indentation     | 2 spaces                           |
| Semicolons      | As needed (omitted where possible) |
| Quotes          | Single quotes (`'`)                |
| JSX quotes      | Single quotes (`'`)                |
| Arrow parens    | Always (`(x) => x`)                |
| End of line     | LF                                 |
| Trailing commas | ES5 (Biome default)                |
| Line length     | Biome/Prettier defaults            |
| File encoding   | UTF-8                              |
| End files with  | A newline                          |

### Editor Setup

The project includes `.vscode/settings.json` that configures:

- **Biome** as the default formatter for JS, TS, JSX, TSX, JSON, JSONC, CSS, GraphQL
- **Prettier** as the default formatter for everything else
- Format on save and paste enabled
- Biome auto-fix and import organization on save

## 5. TypeScript Conventions

### Hand-written code (`tools/`)

- Target: `ES2022`
- Module: `ESNext`
- Module resolution: `node`
- Strict mode enabled
- Use `import type` for type-only imports
- Use `node:` protocol for Node.js builtins (`import { dirname } from 'node:path'`)
- Prefer `abstract class` + inheritance for Nx plugins (see `Plugin` base class)
- No semicolons (enforced by Biome `asNeeded`)

### Generated code (`packages/`)

- Generated by `pulumi-language-nodejs` -- do not edit
- Uses CommonJS-style `require()` in `postinstall.js` scripts
- Uses `declare` keyword for property declarations
- Prettier and Biome are configured to **ignore** these files (see `.prettierignore`)

## 6. JavaScript Conventions (CI Scripts)

Scripts in `.github/scripts/` and `packages/*/scripts/`:

- Use `require('node:...')` with the `node:` protocol for builtins
- Use `spawnSync` (not `exec`) for subprocess calls when possible
- Handle errors with try/catch and `process.exit(1)` on failure
- Use `const` by default; avoid `var`

## 7. Dependency Management

### Package manager

- **pnpm** (version pinned in `mise.toml`)
- Workspace defined in `pnpm-workspace.yaml`
- Lock file: `pnpm-lock.yaml`

### Version pinning

- All dependencies must use **exact versions** (no ranges), enforced by syncpack
- Peer dependencies are excluded from version enforcement
- pnpm overrides are used for security patches on transitive dependencies

### Syncpack

- `syncpack lint` validates version consistency across the monorepo
- `syncpack format` enforces a consistent `package.json` field order (defined in `.syncpackrc.json`)
- Fields are sorted: `name`, `description`, `version`, ... `dependencies`, `devDependencies`, `peerDependencies`

### Supply chain security

- **Aikido Safe Chain** is installed in all CI workflows
- **pnpm `minimumReleaseAge`** is set to 2880 minutes (48 hours), matching the Aikido Safe Chain default
- **Renovate** is configured with `security:minimumReleaseAgeNpm` preset
- `pnpm-workspace.yaml` `allowBuilds` explicitly lists packages permitted to run install scripts

## 8. Linting and Formatting

The project uses a multi-tool pipeline orchestrated by **Nx**:

| Tool               | Purpose                               | Scope                                      |
| ------------------ | ------------------------------------- | ------------------------------------------ |
| Biome              | Linting + formatting (JS/TS/JSON/CSS) | `docs/`                                    |
| Prettier           | Formatting (YAML, Markdown, HTML)     | Workspace-wide (except generated packages) |
| Syncpack           | `package.json` consistency            | Workspace-wide                             |
| TypeScript (`tsc`) | Type checking                         | `tools/`, `packages/` (via postinstall)    |

### Running checks

```bash
npx nx run-many -t check    # Run all checks
npx nx affected -t check    # Check only affected projects
npx nx run-many -t fix       # Auto-fix all fixable issues
```

## 9. Commit and Release Practices

### Commits

- Use **conventional commits**: `type(scope): description`
- Common types: `chore`, `feat`, `fix`, `docs`, `ci`, `build`
- Common scopes: `release`, `deps`, `security`, provider names
- One logical change per commit
- Automated commits follow: `chore(release): release packages`, `chore(release): update packages to match upstream terraform providers`

### Changesets

- Use **Changesets** (`@changesets/cli`) for versioning and changelogs
- Create changesets for user-facing changes: `pnpm changeset`
- Stage changeset files: `git add .changeset/`
- Changesets are published via the Publish workflow

### Releases

- Managed by `changesets/action` in the Publish workflow
- Published to npm with `pnpm release` (`nx run-many -t build && changeset publish`)
- All packages use `"access": "public"` in `publishConfig`

## 10. CI/CD Workflows

| Workflow      | Trigger                    | Purpose                                        |
| ------------- | -------------------------- | ---------------------------------------------- |
| `test.yml`    | Push to main, PRs          | Lint, build, check affected projects           |
| `autofix.yml` | Push to main, PRs          | Auto-fix formatting and dependencies           |
| `publish.yml` | Push to main, after Update | Build and publish packages to npm              |
| `update.yml`  | Weekly (Monday 00:00 UTC)  | Check upstream Terraform providers for updates |

### Workflow conventions

- All actions are **pinned by SHA** (not tags)
- Top-level `permissions: contents: read` with job-level escalation
- pnpm store and Nx cache are cached between runs
- Aikido Safe Chain is installed before `pnpm install`

## 11. Adding a New Provider

New providers are added automatically by the update workflow, but if adding manually:

1. Run `pulumi package add terraform-provider <registry-url> <version>` in a temp project
2. Copy the generated SDK to `packages/<provider-name>/`
3. Create a `package.json` following the existing pattern (see any existing package)
4. Ensure `postinstall.js` is present in `scripts/`
5. Add a changeset describing the new provider

## 12. Testing

- The project currently validates via `tsc` compilation and lint checks
- No unit test framework is configured for the generated provider code
- CI runs `pnpm audit` for dependency vulnerability scanning

## 13. Changes to This Guide

Style evolves. Propose improvements by opening an issue or submitting a PR updating this document.
